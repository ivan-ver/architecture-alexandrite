# Мотивация
| Наименование                      | Описание                                                              |
|-----------------------------------|-----------------------------------------------------------------------|
| Полная видимость бизнес-процессов | Трекинг 100% заказов от создания до доставки                          |
|                                   | Анализ конверсии на каждом этапе воронки                              |
|                                   | Выявление точек трения где клиенты бросают заказы                     |
| Снижение потерь от простоев       | Мгновенное оповещение о проблемах до того, как пожалуются клиенты     |
|                                   | Сокращение времени простоя с часов до минут                           |
|                                   | Предсказание проблем до их возникновения                              |
| Ускорение разработки и релизов    | Автоматическое тестирование производительности после каждого деплоя   |
|                                   | Data-driven решения что именно оптимизировать в первую очередь        |
|                                   | Снижение рисков при выпуске новых фич                                 |
| Оптимизация ресурсов и затрат     | Обоснование масштабирования на реальных метриках, а не предположениях |
|                                   | Выявление неэффективных процессов в производственной цепочке          |
|                                   | Снижение затрат на поддержку за счёт proactive подхода                |

# Выбор подхода к мониторингу
| Компонент                                           | Основной подход   | Дополнительные метрики   | Причина                                                         |
|-----------------------------------------------------|-------------------|--------------------------|-----------------------------------------------------------------|
| Пользовательские сервисы (Online Shop, CRM, MES UI) | Основной подход   | 4 золотых сигнала        | Эти сервисы непосредственно влияют на пользовательский опыт     |
| Backend API (Shop API, CRM API, MES API)            | RED               | USE для ресурсов         | API являются критичными точками интеграции и бизнес-логики      |
| Базы данных                                         | USE               | Запросы/транзакции (RED) | Лучше всего выявляет узкие места и ограничения ресурсов         |
| Очередь                                             | USE               | Длина очереди сообщений  | Для системных ресурсов важно понимать утилизацию и насыщенность |
| MES (расчёты)                                       | RED + кастомные   | USE для ресурсов         | Уникальные характеристики, требующие особого подхода            |
| Бизнес-процессы                                     | Кастомные метрики | -                        | Не покрываются стандартными методологиями, но критически важны  |

# Метрики
| Метрика                                            | Зачем нужна                                                                                                | Ярлыки (labels)                  |
|----------------------------------------------------|------------------------------------------------------------------------------------------------------------|----------------------------------|
| Number of dead-letter-exchange letters in RabbitMQ | Показывает количество неудачно обработанных сообщений. Важно для выявления проблем в потребителях очередей | `queue_name`, `reason`           |
| Number of message in flight in RabbitMQ            | Отслеживает нагрузку на очередь и потенциальные задержки в обработке сообщений                             | `queue_name`                     |
| Number of requests (RPS) for internet shop API     | Показывает нагрузку на основной API магазина. Критично для масштабирования                                 | `endpoint`, `method`             |
| Number of requests (RPS) for CRM API               | Отражает активность продавцов в CRM системе                                                                | `endpoint`, `method`             |
| Number of requests (RPS) for MES API               | Показывает нагрузку на API производства, особенно важен для расчёта стоимости                              | `endpoint`, `method`             |
| CPU % for shop API                                 | Утилизация процессора сервиса магазина. Показывает необходимость масштабирования                           | `instance`, `service`            |
| CPU % for CRM API                                  | Загрузка процессора CRM сервиса. Важно для стабильности работы продавцов                                   | `instance`, `service`            |
| CPU % for MES API                                  | Утилизация процессора MES сервиса. Критично для расчёта стоимости заказов                                  | `instance`, `service`            |
| Memory Utilisation for shop API                    | Использование памяти сервисом магазина. Помогает выявлять утечки памяти                                    | `instance`, `service`            |
| Memory Utilisation for CRM API                     | Использование памяти CRM сервисом. Важно для стабильности                                                  | `instance`, `service`            |
| Memory Utilisation for MES API                     | Использование памяти MES сервисом. Критично для ресурсоёмких расчётов                                      | `instance`, `service`            |
| Memory Utilisation for shop db instance            | Использование памяти основной БД. Показывает необходимость оптимизации запросов                            | `database`, `instance_type`      |
| Memory Utilisation for MES db instance             | Использование памяти БД MES. Важно для производственных процессов                                          | `database`, `instance_type`      |
| Number of connections for shop db instance         | Количество подключений к основной БД. Показывает нагрузку и лимиты                                         | `database`, `user`               |
| Number of connections for MES db instance          | Количество подключений к БД MES. Важно для планирования capacity                                           | `database`, `user`               |
| Response time (latency) for shop API               | Время ответа API магазина. Непосредственно влияет на пользовательский опыт                                 | `endpoint`, `method, percentile` |
| Response time (latency) for CRM API                | Время ответа CRM API. Влияет на эффективность работы продавцов                                             | `endpoint`, `method, percentile` |
| Response time (latency) for MES API                | Время ответа MES API. Критично для расчёта стоимости заказов                                               | `endpoint`, `method, percentile` |
| Size of S3 storage                                 | Объём хранилища 3D-файлов. Помогает планировать затраты на хранение                                        | `bucket`, `file_type`            |
| Size of shop db instance                           | Размер основной БД. Важно для планирования бэкапов и масштабирования                                       | `database`                       |
| Size of MES db instance                            | Размер БД MES. Помогает в управлении производственными данными                                             | `database`                       |
| Number of HTTP 200 for shop API                    | Количество успешных запросов. Показывает доступность сервиса                                               | `endpoint`, `method`             |
| Number of HTTP 500 for shop API                    | Количество ошибок сервера. Критично для стабильности системы                                               | `endpoint`, `method, error_type` |
| Number of HTTP 500 for CRM API                     | Ошибки CRM API. Влияют на работу продавцов                                                                 | `endpoint`, `method, error_type` |
| Number of HTTP 500 for MES API                     | Ошибки MES API. Критичны для производственного процесса                                                    | `endpoint`, `method, error_type` |
| Number of simultanious sessions for shop API       | Количество активных сессий. Показывает нагрузку на аутентификацию                                          | `user_type`, `auth_method`       |
| Kb transferred (received) for shop API             | Входящий трафик. Помогает в анализе нагрузки на сеть                                                       | `endpoint`, `method`             |
| Kb provided (sent) for shop API                    | Исходящий трафик. Важно для планирования bandwidth                                                         | `endpoint`, `method`             |

# План действий
### 1. Развертывание инфраструктуры мониторинга
* Задача 1.1: Создать stack мониторинга на базе Prometheus
  - Развернуть Prometheus в кластере Kubernetes или на выделенных ВМ
  - Настроить high availability режим для отказоустойчивости
  - Реализовать long-term storage (Thanos/Cortex) для хранения метрик на 1+ год
* Задача 1.2: Развернуть Grafana для визуализации
  - Установить Grafana с подключением к Prometheus
  - Настроить SSO аутентификацию для команды
  - Реализовать dashboard provisioning как код
* Задача 1.3: Настроить систему оповещений
  - Развернуть Alertmanager для управления нотификациями
  - Интегрировать с Slack, Telegram и email-уведомлениями
  - Настроить routing и группировку алертов
### 2. Инструментирование приложений
* Задача 2.1: Внедрить метрики в Java-сервисы (Shop API, CRM API)
  - Добавить Prometheus client в Spring Boot приложения
  - Настроить сбор HTTP-метрик (RPS, latency, errors)
  - Реализовать business metrics (заказы, статусы, расчеты)
* Задача 2.2: Инструментировать C# MES API
  - Интегрировать Prometheus.NET библиотеку
  - Настроить метрики расчета стоимости и обработки 3D-файлов
  - Реализовать health checks и readiness probes
* Задача 2.3: Настроить метрики фронтенд-приложений
  - Внедрить Frontend monitoring (Web Vitals, RUM)
  - Настроить сбор ошибок клиентского кода
  - Реализовать tracking бизнес-событий
### 3. Мониторинг инфраструктуры
* Задача 3.1: Настроить мониторинг баз данных
  - Установить PostgreSQL exporters для всех инстансов БД
  - Настроить сбор критичных метрик (connections, queries, locks)
  - Реализовать дашборды для DBA и разработчиков
* Задача 3.2: Мониторинг очереди сообщений
  - Развернуть RabbitMQ exporter
  - Настроить отслеживание очередей, consumers и ошибок
  - Реализовать алерты на dead letters и задержки
* Задача 3.3: Мониторинг хранилища S3
  - Настроить CloudWatch метрики для S3 buckets
  - Реализовать отслеживание размера и количества объектов
  - Настроить алерты на аномальный рост
### 4. Логирование и трейсинг
* Задача 4.1: Внедрить централизованное логирование
  - Развернуть Elastcsearch для хранения логов
  - Настроить log collection со всех сервисов
  - Реализовать log shipping из контейнеров и ВМ
* Задача 4.2: Настроить распределенный трейсинг
  - Внедрить Jaeger для трейсинга
  - Инструментировать критичные цепочки запросов
  - Настроить correlation ID между сервисами
### 5. Бизнес-мониторинг
* Задача 5.1: Реализовать дашборды бизнес-метрик
  - Создать Executive Dashboard с KPI бизнеса
  - Настроить воронку продаж и конверсии
  - Реализовать отслеживание SLA по заказам
* Задача 5.2: Настроить алертинг по бизнес-событиям
  - Реализовать алерты на аномалии в заказах
  - Настроить уведомления о простоях производства
  - Создать reporting для бизнес-аналитики
### 6. Безопасность и доступы
* Задача 6.1: Настроить RBAC для мониторинга
  - Реализовать ролевую модель доступа к дашбордам
  - Настроить разграничение прав между командами
  - Внедрить аудит изменений конфигурации
* Задача 6.2: Обеспечить безопасность инфраструктуры
  - Настроить TLS/SSL для всех компонентов
  - Реализовать изоляцию сетей мониторинга
  - Настроить backup и DR процедуры
